# D5. Context Management & Reliability (15%)

## 핵심 개념

### 5.1 긴 상호작용에서 핵심 정보를 보존하는 컨텍스트 관리
- 점진적 요약(progressive summarization)은 숫자·퍼센트·날짜·고객이 말한 기대치를 모호하게 뭉뚱그릴 위험이 있음
- "lost in the middle" 효과: 모델은 긴 입력의 시작과 끝은 안정적으로 처리하지만 중간 섹션의 내용을 놓칠 수 있음
- 도구 결과가 컨텍스트에 누적되며 실제 관련성에 비해 토큰을 불균형하게 소비(예: 주문 조회에서 5개만 필요한데 40개+ 필드가 쌓임)
- 후속 API 요청에 전체 대화 히스토리를 전달하는 것이 대화의 일관성 유지에 중요
- 해법: 거래 사실(금액·날짜·주문번호·상태)을 요약된 히스토리 바깥의 영구적인 "case facts" 블록으로 따로 추출·유지, 장황한 도구 출력은 관련 필드만 남기고 트리밍, 핵심 발견 요약을 입력 앞쪽에 배치하고 명시적 섹션 헤더로 구조화

### 5.2 효과적인 에스컬레이션과 모호성 해소 패턴 설계
- 적절한 에스컬레이션 트리거: 고객의 명시적인 사람 연결 요청, 정책 예외/공백(단순히 복잡한 케이스가 아님), 의미 있는 진전이 불가능한 상황
- 고객이 명시적으로 요구할 때는 즉시 에스컬레이션, 이슈가 단순하면 먼저 해결을 제안
- 감정(sentiment) 기반 에스컬레이션이나 모델의 자가 보고 confidence 점수는 실제 케이스 복잡도를 신뢰성 있게 대변하지 못함
- 동일 식별자로 여러 고객이 매칭되면 휴리스틱으로 임의 선택하지 말고 추가 식별 정보를 요청해야 함

### 5.3 멀티에이전트 시스템 전반의 에러 전파 전략
- 실패 유형, 시도한 쿼리, 부분 결과, 가능한 대안을 포함한 구조화된 에러 컨텍스트가 코디네이터의 지능적인 복구 판단을 가능케 함
- 접근 실패(타임아웃 등 재시도 판단 필요)와 정상적인 빈 결과(매칭 없는 성공적 쿼리)는 반드시 구분해서 보고
- 서브에이전트는 transient 실패에 대해 로컬 복구를 시도하고, 해결할 수 없는 에러만 시도 내역·부분 결과와 함께 상위로 전파
- 에러를 침묵 처리(빈 결과를 성공으로 위장)하거나 단일 실패로 전체 워크플로우를 종료하는 것은 둘 다 안티패턴
- synthesis 출력에는 어떤 발견이 충분히 뒷받침되고 어떤 토픽 영역에 소스 부재로 인한 공백이 있는지 커버리지 주석을 포함

### 5.4 대규모 코드베이스 탐색에서 컨텍스트를 효과적으로 관리
- 장시간 세션의 컨텍스트 저하: 일관되지 않은 답을 주거나, 앞서 발견한 구체적 클래스 대신 "전형적인 패턴"을 언급하기 시작하는 현상
- scratchpad 파일에 핵심 발견을 기록해 컨텍스트 경계를 넘어 일관성을 유지
- 장황한 탐색 출력은 서브에이전트에 위임해 격리하고, 메인 에이전트는 고수준 코디네이션만 담당(예: "모든 테스트 파일 찾기", "환불 흐름 의존성 추적")
- 크래시 복구를 위해 각 에이전트가 알려진 위치에 상태를 내보내고, 코디네이터가 재개 시 manifest를 로드해 prompt에 주입
- `/compact`로 컨텍스트가 장황한 탐색 출력으로 가득 찰 때 사용량을 줄임

### 5.5 휴먼 리뷰 워크플로우와 confidence 보정 설계
- 종합 정확도 지표(예: 97% 전체 정확도)가 특정 문서 유형이나 필드의 저성능을 가릴 수 있음
- stratified random sampling으로 고신뢰 추출의 에러율을 지속적으로 측정하고 새로운 에러 패턴을 탐지
- 자동화를 확대하기 전에 문서 유형·필드 세그먼트별 정확도를 분석해 일관된 성능을 검증해야 함
- 필드 단위 confidence 점수를 라벨링된 검증셋으로 보정해 리뷰 우선순위를 정함
- confidence가 낮거나 소스 문서가 모호/상충하는 추출은 제한된 리뷰어 역량을 우선 배분해 사람 리뷰로 라우팅

### 5.6 정보 출처(provenance) 보존과 다중 소스 종합의 불확실성 처리
- 발견을 압축할 때 출처 귀속(attribution)이 손실되기 쉬움 — claim과 source의 매핑을 구조화해서 보존·병합해야 함
- 신뢰할 수 있는 소스 간 통계가 충돌하면 임의로 하나를 선택하지 말고, 출처 귀속과 함께 충돌을 주석 처리
- 발행/수집 날짜를 구조화된 출력에 요구해 시간차로 인한 값의 차이를 모순으로 잘못 해석하는 것을 방지
- 잘 정립된 발견과 다투고 있는(contested) 발견을 구분하는 명시적 섹션으로 보고서를 구조화, 원본 소스의 성격과 방법론적 맥락을 보존
- 콘텐츠 유형에 맞게 다른 형식으로 렌더링(금융 데이터는 표, 뉴스는 prose, 기술적 발견은 구조화된 리스트) — 모든 것을 하나의 균일한 포맷으로 강제하지 않음
