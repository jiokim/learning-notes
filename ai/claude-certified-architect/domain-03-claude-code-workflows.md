# D3. Claude Code Configuration & Workflows (20%)

## 핵심 개념

### 3.1 CLAUDE.md 계층 구성, 스코핑, 모듈화
- 계층: 사용자 레벨(`~/.claude/CLAUDE.md`), 프로젝트 레벨(`.claude/CLAUDE.md` 또는 루트 `CLAUDE.md`), 디렉터리 레벨(서브디렉터리 `CLAUDE.md`)
- 사용자 레벨 설정은 그 사용자에게만 적용 — 버전 관리로 팀원과 공유되지 않음
- `@import` 구문으로 외부 파일을 참조해 CLAUDE.md를 모듈식으로 유지(예: 패키지별 표준 파일 import)
- `.claude/rules/` 디렉터리로 토픽별 규칙 파일을 분리 — 모놀리식 CLAUDE.md의 대안
- `/memory` 명령으로 현재 로드된 메모리 파일을 확인해 세션 간 불일치를 진단

### 3.2 커스텀 슬래시 커맨드와 스킬 구성
- 프로젝트 스코프 커맨드(`.claude/commands/`, 버전 관리로 팀 공유) vs 사용자 스코프 커맨드(`~/.claude/commands/`, 개인용)
- `.claude/skills/`의 `SKILL.md`는 `context: fork`, `allowed-tools`, `argument-hint` 같은 frontmatter를 지원
- `context: fork`는 스킬을 격리된 서브에이전트 컨텍스트에서 실행해 출력이 메인 대화를 오염시키지 않게 함(예: 장황한 코드베이스 분석)
- `allowed-tools`로 스킬 실행 중 도구 접근을 제한(예: 파일 쓰기만 허용해 파괴적 동작 방지)
- `argument-hint`로 인자 없이 호출됐을 때 필요한 파라미터를 안내
- 개인 변형은 `~/.claude/skills/`에 다른 이름으로 만들어 팀원에게 영향 없이 커스터마이즈
- 스킬(필요할 때 온디맨드 호출) vs CLAUDE.md(항상 로드되는 범용 표준) 용도 구분

### 3.3 경로별(path-specific) 규칙으로 조건부 컨벤션 로딩
- `.claude/rules/` 파일은 YAML frontmatter의 `paths` 필드에 glob 패턴을 지정(예: `paths: ["**/*.test.tsx"]`)
- 매칭되는 파일을 편집할 때만 규칙이 로드되어 불필요한 컨텍스트·토큰 사용을 줄임
- 코드 옆에 흩어진 테스트 파일처럼 여러 디렉터리에 걸친 컨벤션에는 디렉터리별 CLAUDE.md보다 path-specific 규칙이 유리

### 3.4 Plan 모드 vs 직접 실행 판단
- Plan 모드: 대규모 변경, 여러 유효한 접근법, 아키텍처 결정, 다중 파일 수정이 필요한 복잡한 작업에 적합
- 직접 실행: 범위가 명확하고 단순한 변경(예: 단일 함수에 검증 추가)에 적합
- Plan 모드는 변경을 커밋하기 전에 안전하게 코드베이스를 탐색·설계할 수 있게 해 비용이 큰 재작업을 예방
- Explore 서브에이전트로 장황한 탐색 결과를 분리해 요약만 메인 대화에 반환 — 멀티페이즈 작업의 컨텍스트 고갈 방지

### 3.5 점진적 개선(iterative refinement) 기법
- 산문 설명이 일관되게 해석되지 않을 때, 구체적인 입출력 예시 2~3개가 기대 변환을 전달하는 가장 효과적인 방법
- 테스트 주도 반복: 기대 동작·엣지 케이스·성능 요구사항을 다루는 테스트 스위트를 먼저 작성하고, 실패를 공유하며 개선
- interview 패턴: Claude가 먼저 질문하게 해서 개발자가 미처 고려하지 못한 사항(캐시 무효화 전략, 실패 모드 등)을 표면화
- 서로 영향을 주는 이슈는 한 메시지에 모아서 전달, 독립적인 이슈는 순차적으로 반복 처리

### 3.6 Claude Code를 CI/CD 파이프라인에 통합
- `-p`(`--print`) 플래그로 비대화형 모드 실행 — 인터랙티브 입력을 기다리지 않고 자동화 파이프라인에서 행(hang) 방지
- `--output-format json` + `--json-schema`로 CI에서 머신이 파싱 가능한 구조화된 산출물 보장(예: PR 인라인 코멘트로 자동 게시)
- CLAUDE.md로 CI에서 호출되는 Claude에 테스트 표준·픽스처·리뷰 기준 같은 프로젝트 컨텍스트 제공
- 새 커밋에 대해 리뷰를 다시 실행할 때는 이전 리뷰 결과를 컨텍스트에 포함해 새/미해결 이슈만 보고하도록 지시(중복 코멘트 방지)
- 코드를 생성한 동일 세션은 자기 검토에 약함 — 독립된 리뷰 인스턴스가 추론 컨텍스트 없이 더 미세한 이슈를 잘 잡음
