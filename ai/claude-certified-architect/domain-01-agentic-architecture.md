# D1. Agentic Architecture & Orchestration (27%)

## 핵심 개념

### 1.1 에이전트 루프 설계
- `stop_reason`이 `"tool_use"`면 도구 실행 후 결과를 대화 히스토리에 추가하고 반복, `"end_turn"`이면 종료
- 다음 행동은 모델이 컨텍스트 기반으로 결정 — 사전 정의된 의사결정 트리나 고정된 도구 시퀀스가 아님
- 안티패턴: 자연어로 종료 신호를 파싱하거나, 임의의 반복 횟수 제한을 주 종료 조건으로 쓰거나, 어시스턴트 텍스트 존재 여부로 완료를 판단

### 1.2 코디네이터-서브에이전트 오케스트레이션
- Hub-and-spoke 구조: 코디네이터가 서브에이전트 간 모든 통신·에러 처리·라우팅을 담당
- 서브에이전트는 격리된 컨텍스트로 동작 — 코디네이터의 대화 히스토리를 자동으로 물려받지 않음
- 코디네이터는 쿼리 복잡도를 분석해 어떤 서브에이전트를 호출할지 동적으로 결정해야 함
- 너무 좁게 작업을 분해하면 넓은 토픽의 일부 영역이 통째로 누락될 위험이 있음

### 1.3 서브에이전트 호출 · 컨텍스트 전달 · 스폰
- `Task` 도구가 서브에이전트를 스폰하는 메커니즘 — 코디네이터의 `allowedTools`에 `"Task"`가 포함돼야 함
- 서브에이전트는 자동으로 부모 컨텍스트를 상속하거나 메모리를 공유하지 않음 — 필요한 컨텍스트는 prompt에 명시적으로 포함해야 함
- `AgentDefinition`으로 각 서브에이전트 타입의 description, 시스템 프롬프트, 도구 제한을 구성
- 여러 `Task` 호출을 한 코디네이터 응답에서 동시에 emit하면 병렬 실행됨 (여러 턴에 나눠 호출하면 순차 실행)
- `fork_session`으로 공유된 분석 기준점에서 발산하는 여러 접근법을 탐색 가능

### 1.4 멀티스텝 워크플로우의 enforcement와 handoff
- 프로그래매틱 강제(hooks, prerequisite gate)와 prompt 기반 가이드는 다름 — 결정론적 준수가 필요하면(예: 환불 전 신원 확인) 후자만으로는 실패율이 0이 아님
- 에스컬레이션 시 고객 정보·근본 원인·권장 조치를 포함한 구조화된 handoff 프로토콜 필요
- 멀티 컨선(multi-concern) 요청은 항목별로 분해해 각각 조사 후 통합된 응답으로 합성

### 1.5 Agent SDK hooks
- `PostToolUse` 같은 hook 패턴으로 모델이 처리하기 전에 도구 결과를 가로채 변환/정규화 (예: 서로 다른 포맷의 타임스탬프 통일)
- 도구 호출 자체를 가로채는 hook으로 정책 위반(예: 임계값 초과 환불)을 차단하고 대안 워크플로우로 리다이렉트
- 결정론적 보장이 필요한 비즈니스 규칙은 hooks, 확률적 준수로 충분하면 prompt 지시로 처리

### 1.6 작업 분해 전략
- 고정 순차 파이프라인(prompt chaining: 파일별 분석 → cross-file 통합 패스)과 동적 적응형 분해(발견 내용에 따라 서브태스크 생성)를 구분해서 선택
- 예측 가능한 다측면 리뷰는 prompt chaining, 열린 결말의 탐색 작업은 동적 분해가 적합
- 열린 작업(예: "레거시 코드베이스에 테스트 추가")은 구조 파악 → 고영향 영역 식별 → 발견에 따라 적응하는 우선순위 계획 순으로 분해

### 1.7 세션 상태 · 재개 · 포크
- `--resume <session-name>`으로 특정 이전 대화를 이름으로 재개
- `fork_session`으로 공유된 분석 기준점에서 독립적인 분기를 생성해 서로 다른 접근법 비교
- 코드가 수정된 후 세션을 재개할 때는 변경된 파일을 명시적으로 알려줘야 함
- 이전 도구 결과가 stale하다면, 재개보다 구조화된 요약을 주입한 새 세션이 더 안정적
