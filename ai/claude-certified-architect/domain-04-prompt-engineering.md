# D4. Prompt Engineering & Structured Output (20%)

## 핵심 개념

### 4.1 명시적 기준으로 정밀도를 높이고 거짓양성을 줄이는 프롬프트 설계
- "보수적으로 판단해라", "고신뢰 발견만 보고해라" 같은 모호한 일반 지시는 confidence 기반 필터링과 달리 정밀도를 실질적으로 개선하지 못함
- 어떤 이슈를 보고하고(버그·보안) 어떤 것을 건너뛸지(마이너 스타일, 로컬 패턴) 구체적인 범주형 기준으로 정의해야 함
- 거짓양성률이 높은 카테고리는 개발자의 신뢰를 깎아먹음 — 우선 그 카테고리를 비활성화해 신뢰를 회복한 뒤 프롬프트를 개선
- 구체적인 코드 예시를 곁들인 명시적 심각도 기준을 정의하면 일관된 분류가 가능해짐

### 4.2 Few-shot 프롬프팅으로 출력 일관성과 품질 향상
- 상세한 지시만으로 결과가 일관되지 않을 때, few-shot 예시가 일관된 포맷·실행 가능한 출력을 얻는 가장 효과적인 기법
- 모호한 케이스(도구 선택, 브랜치 단위 테스트 커버리지 갭 등)의 처리 방식을 예시로 보여주면 모델이 사전에 명시되지 않은 새로운 패턴에도 일반화 가능
- 비정형 측정값, 다양한 문서 구조 등 추출 작업의 환각(hallucination) 감소에도 효과적
- 2~4개의 타겟형 예시로 왜 한 행동이 다른 그럴듯한 대안보다 선택됐는지 보여주는 것이 핵심

### 4.3 tool_use와 JSON Schema로 구조화된 출력 강제
- `tool_use` + JSON Schema는 스키마를 준수하는 구조화된 출력을 보장하는 가장 신뢰도 높은 방법 — JSON 문법 오류를 원천적으로 제거
- `tool_choice`: `"auto"`(텍스트 반환 가능) / `"any"`(반드시 도구 호출, 어떤 것이든) / 특정 도구 강제 지정
- 엄격한 JSON Schema는 문법 오류는 막아주지만, 의미적 오류(라인 항목 합계 불일치, 값이 잘못된 필드에 들어가는 것)까지는 막아주지 못함
- 스키마 설계 시 고려사항: 필수 vs 선택 필드, "other" + 세부 설명 문자열 패턴으로 확장 가능한 카테고리 설계
- 소스 문서에 정보가 없을 수 있는 필드는 nullable(optional)로 설계해 모델이 값을 지어내지 않도록 함

### 4.4 검증 · 재시도 · 피드백 루프로 추출 품질 확보
- retry-with-error-feedback: 재시도 시 구체적인 검증 에러를 프롬프트에 추가해 모델의 자기 수정을 유도
- 재시도의 한계: 필요한 정보가 소스 문서에 단순히 없을 때는 재시도가 효과 없음(포맷·구조적 오류일 때만 효과적)
- `detected_pattern` 같은 필드로 어떤 코드 구성이 발견을 유발했는지 추적해 거짓양성 패턴을 체계적으로 분석
- semantic validation 오류(값 불일치, 잘못된 필드 배치)와 schema syntax 오류(tool_use로 이미 제거됨)는 다른 문제

### 4.5 효율적인 배치 처리 전략 설계
- Message Batches API: 50% 비용 절감, 최대 24시간 처리 윈도우, 지연시간(latency) SLA 보장 없음
- 배치 처리는 논블로킹·지연 허용 워크로드(야간 리포트, 주간 감사, 야간 테스트 생성)에 적합 — 블로킹 워크플로우(pre-merge 체크)엔 부적합
- Batch API는 단일 요청 안에서 멀티턴 도구 호출(중간에 도구를 실행하고 결과를 받는 것)을 지원하지 않음
- `custom_id` 필드로 배치 요청/응답 쌍을 매칭하고, 실패한 문서만 식별해 수정 후 재제출

### 4.6 멀티 인스턴스 · 멀티 패스 리뷰 아키텍처 설계
- 같은 세션에서의 자가 검토(self-review)는 생성 시점의 추론 컨텍스트를 유지하기 때문에 자기 결정을 의심할 가능성이 낮음
- 이전 추론 컨텍스트가 없는 독립적인 리뷰 인스턴스가 self-review 지시나 extended thinking보다 미세한 이슈를 더 잘 잡음
- 멀티 패스 리뷰: 큰 리뷰를 파일별 로컬 분석 패스 + 별도의 cross-file 통합 패스로 분리해 attention dilution과 모순된 결과를 방지
