# Chapter 6. Tackling Complex Business Logic

## 핵심 개념

### 도메인 모델 (Domain Model) 개요
Domain Model 패턴은 원래 Martin Fowler의 «Patterns of Enterprise Application Architecture»에서 처음 소개됐고, Eric Evans는 거기서 한 발 더 나가 코드를 비즈니스 도메인의 모델에 더 밀접하게 연결시키는 구체적인 패턴들 — aggregate, value object, repository 등 — 을 제시했다. 저자가 "tactical domain-driven design"이라는 용어보다 Fowler의 원래 용어인 "domain model"을 고집하는 이유는, "DDD를 실천한다 = 반드시 이 패턴들을 써서 비즈니스 로직을 구현해야 한다"는 오해를 없애기 위해서다 — 패턴의 이름은 "domain model"이고, aggregate와 value object는 그 패턴의 빌딩 블록일 뿐이다.
이 패턴은 CRUD 인터페이스 대신 복잡한 상태 전이, 비즈니스 규칙, 그리고 항상 보호돼야 하는 불변식(invariants)을 다루기 위한 것이다. 헬프데스크 티켓 lifecycle 예시(SLA, escalation, 자동 재배정, 7일 후 자동 종료 등)처럼 규칙들이 서로 뒤엉킨 의존관계의 그물을 이루며 모두 같은 lifecycle 관리 로직에 영향을 줄 때, 이를 Active Record로 구현하면 로직이 여기저기 중복되거나 일부 규칙이 잘못 구현되기 쉬워 시스템 상태를 손상시키기 쉽다.
도메인의 비즈니스 로직은 이미 본질적으로 복잡하므로, 그걸 모델링하는 객체가 거기에 우발적 복잡성(accidental complexity)을 더하면 안 된다 — 그래서 도메인 모델의 객체들은 DB 호출 등 인프라·기술적 관심사를 구현하지 않는 순수한 객체("plain old objects")여야 한다. 기술적 관심사보다 비즈니스 로직을 우선시하는 이 설계는, 도메인 모델의 객체들이 바운디드 컨텍스트의 유비쿼터스 언어 용어를 그대로 따르기 쉽게 만들어 코드가 유비쿼터스 언어를 "말하게" 하고 도메인 전문가의 멘탈 모델을 따르게 한다.

### 값 객체 (Value Object)
값 객체는 자신의 값들의 조합(composition of values)으로 식별되므로 별도의 식별 필드가 필요 없다 — `Color` 클래스에 별도의 `ColorId`를 두면, red/green/blue 값은 같은데 `ColorId`만 다른 두 row가 생길 수 있어 오히려 버그의 원인이 된다. "Primitive obsession"은 비즈니스 도메인의 개념을 표현할 때 string/int 같은 원시 타입에만 전적으로 의존하는 코드 스멜로, `Person` 예시처럼 모든 필드가 원시 타입이고 형식이 관례로만 보장되면 ① 검증 로직이 여기저기 중복되기 쉽고, ② 그 검증이 값이 쓰이기 전에 항상 호출되도록 강제하기 어렵다는 두 가지 설계 위험이 생긴다. 값 객체로 다시 설계하면 검증 로직이 값 객체 자신의 내부(생성자/`Parse` 메서드)로 이동해, 검증이 한 곳에만 있어 중복이 사라지고 그 검증을 통과해야만 인스턴스를 만들 수 있어 호출이 자동으로 보장된다.
값 객체의 필드 중 하나라도 바뀌면 같은 객체의 상태가 바뀐 게 아니라 개념적으로 다른 값이 된 것이므로, 값 객체는 불변(immutable)으로 구현된다 — `MixWith` 같은 메서드가 원본 인스턴스를 바꾸지 않고 새 인스턴스를 반환하는 이유다. 동등성(equality)도 식별자나 참조가 아니라 값들의 조합 자체로 판단하므로 `Equals`나 `==`는 필드 값을 직접 비교하도록 오버라이드해야 한다. 책은 값 객체를 "가능할 때마다(whenever you can)" 쓰라고 권한다 — 코드를 더 표현력 있게 만들고 흩어지기 쉬운 로직을 캡슐화할 뿐 아니라 불변이라 부작용 없이 스레드 안전하기 때문이다. 특히 다른 엔티티의 속성을 표현하는 도메인 요소, 그리고 상태(status)·비밀번호·화폐(money) 같은 값에 값 객체를 쓰지 않으면 반올림 오류 같은 위험한 버그로 이어지기 쉽다.

### 엔티티 (Entity)
엔티티는 값 객체와 정확히 반대되는 개념이다 — 동명이인 예시처럼 이름이 같다고 같은 사람인 건 아니므로, 인스턴스를 구별하려면 명시적인 식별 필드(ID)가 필요하다. 이 식별 필드는 ① 각 인스턴스에 대해 유일(unique)해야 하고, ② 아주 드문 예외를 빼면 엔티티의 생애주기 동안 불변(immutable)으로 유지돼야 한다. 값 객체는 "값 자체"가 정체성이라 값이 바뀌면 다른 값이 되지만, 엔티티는 "ID"가 정체성이므로 그 정체성을 유지한 채로 상태가 변해야(가변이어야) 같은 대상의 생애주기를 표현할 수 있다 — 엔티티가 불변이라면 상태가 하나라도 바뀔 때마다 "다른 객체"가 돼버려 ID로 추적하는 의미가 사라진다. 책이 "entity"를 도메인 모델의 독립적인 빌딩 블록 목록에 넣지 않은 이유도, 엔티티는 독립적으로 구현되지 않고 항상 애그리거트(aggregate) 패턴의 맥락 안에서만 구현되기 때문이다.

### 애그리거트 (Aggregate)
애그리거트는 명시적인 식별 필드가 필요하고 상태가 변한다는 점에서 엔티티이지만, 그 이상이다 — 이 패턴의 목표는 자신의 데이터의 정합성(consistency)을 보호하는 것이다. 애그리거트 자신의 비즈니스 로직만이 그 상태를 변경할 수 있고, 바깥의 모든 프로세스·객체는 읽기(read)만 할 수 있다. 상태를 바꾸는 메서드들은 커맨드(commands)라 부르며, 일반 public 메서드나 파라미터 객체를 `Execute` 메서드에 전달하는 형태로 구현된다.
애그리거트의 상태 변경은 하나의 원자적(atomic) 연산으로 커밋돼야 하며, 부분적으로만 업데이트된 상태가 커밋되는 경우는 절대 없어야 한다. "하나의 데이터베이스 트랜잭션에는 하나의 애그리거트 인스턴스만"이라는 규칙은 설계자가 비즈니스 도메인의 불변식을 제대로 반영해 애그리거트 경계를 신중하게 설계하도록 강제하며, 한 오퍼레이션이 여러 애그리거트를 동시에 커밋해야 한다면 그건 잘못된 트랜잭션 경계(경계를 잘못 그었다는 신호)다. 여러 프로세스가 동시에 같은 애그리거트를 수정하면 나중 트랜잭션이 먼저 커밋된 변경을 모르고 덮어쓸 위험(lost update)이 있어 동시성 관리가 필수인데, 가장 단순한 구현은 버전(version) 필드를 두고 매 업데이트마다 증가시켜 커밋 시 현재 버전이 읽었을 때와 같은지 확인하는 것이다(5장의 낙관적 동시성 제어와 동일한 원리).
"aggregate(집계)"라는 이름은 같은 트랜잭션 경계에 속하는 엔티티와 값 객체들을 한데 모은다는 데서 왔다. 여러 객체가 동시에 수정될 수 있거나 한 객체의 비즈니스 규칙이 다른 객체의 상태에 의존할 때 같은 트랜잭션 경계를 공유해야 하는데, 티켓 재배정 규칙 예시(에이전트가 응답 시간의 50% 내에 새 메시지를 안 읽으면 재배정)에서 `Ticket`의 규칙이 `Message`의 읽음 여부에 직접 의존하므로 둘은 같은 애그리거트 경계 안에 있어야 한다.
애그리거트가 너무 커지면 성능·확장성 문제가 생기므로, 경계를 결정하는 가이딩 원칙은 데이터의 일관성이다 — 비즈니스 로직상 강하게 일관적(strongly consistent)이어야 하는 정보만 애그리거트 안에 포함하고, 최종적 일관성(eventually consistent)으로 충분한 정보는 경계 밖에(다른 애그리거트의 일부 또는 ID 참조로) 둔다. `Ticket` 예시에서 `Messages`는 경계 안에, `Customer`·`Products`·`AssignedAgent`는 ID로만 참조되는 이유도 이 기준에서 나온다 — 메시지의 읽음 여부가 최종적 일관성으로 처리되면 상당수 티켓이 불필요하게 재배정돼 시스템 상태가 손상되지만, 고객·제품·에이전트는 약간 오래된 값을 참조해도 비즈니스 로직이 잘못된 상태로 빠지지 않기 때문이다.
애그리거트가 엔티티들의 계층 구조일 때, 그중 단 하나만 애그리거트의 public interface인 애그리거트 루트(aggregate root)로 지정된다. `AcknowledgeMessage` 예시에서 실제로 바뀌는 값은 `Message`의 `WasRead` 필드지만, 그 연산은 애그리거트 루트인 `Ticket`을 통해서만 호출할 수 있다 — `Message`는 경계 안에 속한 엔티티일 뿐, 독립적인 public interface를 갖지 않기 때문이다.
도메인 이벤트(domain event)는 비즈니스 도메인에서 일어난 중요한 사건을 설명하는 메시지로(예: 티켓 배정됨, 티켓 에스컬레이션됨, 메시지 수신됨), 이미 일어난 일을 기술하므로 이름이 과거 시제로 표현된다. 도메인 이벤트는 애그리거트의 public interface의 일부이며, 애그리거트가 이벤트를 발행(publish)하면 다른 프로세스·애그리거트·외부 시스템이 구독(subscribe)해 각자의 로직을 실행할 수 있다.
마지막으로, 애그리거트의 이름·데이터 필드·액션·도메인 이벤트는 모두 바운디드 컨텍스트의 유비쿼터스 언어를 기준으로 지어져야 한다 — 코드는 개발자들이 서로, 그리고 도메인 전문가와 대화할 때 쓰는 언어를 기반으로 해야 한다는 Eric Evans의 원칙이 여기서도 그대로 적용된다.

### 도메인 서비스 (Domain Service)
어떤 애그리거트나 값 객체에도 속하지 않거나 여러 애그리거트에 걸쳐 관련된 비즈니스 로직은 도메인 서비스로 구현한다. 도메인 서비스는 상태가 없는(stateless) 객체로, 대부분 어떤 계산이나 분석을 수행하기 위해 시스템의 여러 컴포넌트에 대한 호출을 오케스트레이션한다 — `ResponseTimeFrameCalculationService` 예시가 좋은 후보인 이유도, 응답 시한 계산이 티켓 자체의 데이터뿐 아니라 에이전트의 부서 정책(SLA)과 근무 일정까지 여러 소스의 정보를 필요로 하기 때문이다. 도메인 서비스는 "한 트랜잭션에 애그리거트 인스턴스 하나만 수정 가능"이라는 규칙을 피해 가는 우회로가 아니다 — 그 규칙은 여전히 그대로 유지되며, 도메인 서비스는 여러 애그리거트의 데이터를 읽어야 하는 계산 로직을 구현하는 데 적합할 뿐이다. 이름이 비슷해도 마이크로서비스나 SOA 같은 다른 "서비스" 개념과는 무관하다.

### 복잡도 관리 (Managing Complexity)
Eliyahu Goldratt에 따르면 시스템의 복잡도는 그 행동을 제어·예측하기 어려운 정도를 평가하는 것이며, 이는 시스템의 "자유도(degrees of freedom)" — 상태를 기술하는 데 필요한 데이터 포인트의 수 — 로 반영된다. `ClassA`와 `ClassB`는 둘 다 변수가 5개지만, `ClassA`는 다섯 변수가 서로 독립적이라 자유도가 5인 반면 `ClassB`는 B·C·E가 A·D의 함수로 자동 계산돼 A와 D만 알면 나머지를 유도할 수 있어 자유도가 2뿐이다. 자유도가 많을수록 가능한 상태의 조합이 많아져 행동을 예측·제어하기 어려워지므로, 값들 사이에 불변식(관계)이 있어 자유도가 줄어들수록 복잡도는 낮아진다. 애그리거트와 값 객체 패턴이 자유도를 줄이는 방식도 같은 원리다 — 값 객체는 자신의 상태와 관련된 모든 로직을 자기 경계 안에 두고, 애그리거트도 자기 자신의 커맨드를 통해서만 수정되게 함으로써 비즈니스 불변식을 각자의 경계 안에 캡슐화하고 보호해 외부에서 자유도를 임의로 늘릴 수 없게 만든다. 도메인 모델 패턴은 복잡한 비즈니스 로직을 가진 서브도메인에만 적용되므로, 그런 서브도메인은 대부분 핵심 서브도메인(core subdomain) — 소프트웨어의 심장이다.

### 결론
도메인 모델은 값 객체, 애그리거트, 도메인 서비스라는 세 빌딩 블록으로 구성된다 — 값 객체는 자신의 값들로만 식별되는 불변의 도메인 개념이고, 애그리거트는 트랜잭션 경계를 공유하는 엔티티들의 계층 구조로 그 경계 안의 모든 데이터가 강하게 일관적이어야 하며, 도메인 서비스는 애그리거트나 값 객체 어디에도 속하지 않는 로직을 담는 상태 없는 객체다. 객체의 상태를 외부에서 직접 수정할 수 없다는 제약은 관련 비즈니스 로직이 애그리거트·값 객체의 경계 안에서만 구현되도록 강제해, 그 로직이 애플리케이션 계층 등 다른 곳에 중복되는 것을 막아준다. 다음 챕터는 도메인 모델 패턴을 구현하는 더 발전된 방식을 다루는데, 이번에는 시간(time)이라는 차원을 모델의 본질적인 일부로 만든다고 예고한다.

복잡한 비즈니스 로직은 도메인 모델 패턴(값 객체·애그리거트·도메인 서비스)으로 다루는데, 값 객체와 애그리거트는 비즈니스 불변식을 자신의 경계 안에 캡슐화해 자유도를 줄임으로써 복잡도를 낮추고, 애그리거트의 경계는 강한 일관성이 필요한 데이터만 포함하도록(나머지는 ID로만 참조하도록) 설계해야 한다.

## 밑줄 친 문장

> *"The invariants introduced in ClassB reduce its complexity. That's what both aggregate and value object patterns do: encapsulate invariants and thus reduce complexity."*

Goldratt의 "복잡도 = 제어·예측이 어려운 정도, 즉 자유도(degrees of freedom)"라는 정의를 `ClassA`/`ClassB` 예시로 보여준 뒤, 그걸 그대로 애그리거트·값 객체 패턴에 연결하는 문장. 두 패턴이 "왜" 복잡도를 낮추는지를 추상적 설명이 아니라 측정 가능한 개념(자유도)으로 짚어준다는 점이 인상적이다.
