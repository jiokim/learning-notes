# Chapter 3. Managing Domain Complexity

## 핵심 개념

### Inconsistent Models (모델의 불일치)
같은 회사 안에서도 도메인 전문가들의 정신 모델은 부서마다 다를 수 있다. 텔레마케팅 회사 예시에서 '리드(lead)'는 마케팅 부서에서는 "잠재고객이 관심을 보였다는 알림 이벤트"지만, 영업 부서에서는 "영업 프로세스의 전체 수명주기를 나타내는 장기 프로세스"다.
이런 모호함은 사람 간 대화에서는 맥락으로 쉽게 추론되어 문제가 안 되지만, 소스 코드는 모호성에 잘 대처하지 못해 소프트웨어로 구현할 때는 문제가 된다.
이를 해결하려는 두 가지 잘못된 접근이 있다. 모든 것을 아우르는 거대한 통합 모델(전사 ERD)은 불필요한 정보를 걸러내는 복잡함, 필요한 것을 찾는 복잡함, 데이터 일관성을 유지하는 복잡함과 끝없이 씨름하게 되어 결국 어디에도 효과적이지 않다. 용어에 접두사를 붙이는 방법("marketing lead", "sales lead")은 인지부하가 크고, 실제 대화에서는 아무도 그렇게 부르지 않아 구현이 유비쿼터스 언어와 어긋난다.

### Bounded Context (바운디드 컨텍스트)
바운디드 컨텍스트는 세분화된 유비쿼터스 언어 각각이 일관성을 띠게 하는 경계다. 유비쿼터스 언어를 여러 개의 더 작은 언어로 나누고, 각각이 적용되는 명시적 범위를 정하는 것이다.
모델은 경계 없이는 존재할 수 없다 — 모델의 본질은 목적에 안 맞는 것을 버리는 선택적 추상화인데, 버릴 기준(경계)이 없으면 아무것도 버리지 못해 결국 현실 전체를 끌어안게 되고, 더 이상 모델이 아니라 현실의 복사본이 되어버린다.
유비쿼터스 언어가 "ubiquitous(보편적)"하다는 것도 조직 전체에서 통용된다는 뜻이 아니라, 자신의 바운디드 컨텍스트 안에서만 예외 없이 일관되게 적용된다는 뜻이다. 컨텍스트를 벗어나면 같은 용어가 다른 의미를 가져도 문제가 되지 않는다.

### Bounded Context의 크기 (Scope)
바운디드 컨텍스트가 넓을수록 일관성 유지가 어렵고, 좁을수록 통합 오버헤드가 늘어나는 트레이드오프가 있다. 크기 자체로 좋고 나쁨이 정해지지 않으며, 모델 평가 기준과 마찬가지로 실제 비즈니스 문제와 조직 상황(팀 구성, 독립적 배포·확장 필요성 등)에 맞게 쓸모 있는지가 유일한 기준이다.

### Bounded Context vs Subdomain
이 챕터의 핵심 구분: Subdomain은 발견되고(discovered), Bounded Context는 설계된다(designed). Subdomain의 경계는 회사의 비즈니스 전략이 이미 정해놓은 것이라 분석해서 발견할 뿐이고, Bounded Context의 경계는 엔지니어링 팀이 소프트웨어 설계 차원에서 직접 긋는 것이다.
Subdomain만 있으면 구현 단위가 정해지지 않고, Bounded Context만 있으면 어디에 엔지니어링 역량을 집중해야 하는지에 대한 비즈니스 구조를 놓치기 때문에 두 개념이 모두 필요하다. 둘은 항상 1:1로 매칭되지 않으며, 하나의 Bounded Context에 여러 Subdomain이 들어갈 수도 있다.

### Boundaries: Physical & Ownership
바운디드 컨텍스트는 두 가지 경계를 만든다. 물리적 경계(독립된 서비스/프로젝트 단위로 구현·배포·버전관리)와 소유 경계(한 팀이 책임지는 단위)다.
팀과 바운디드 컨텍스트의 관계는 단방향이다 — 한 팀은 여러 바운디드 컨텍스트를 소유할 수 있지만, 하나의 바운디드 컨텍스트는 오직 한 팀만 소유해야 한다. 바운디드 컨텍스트 안에 여러 Subdomain이 있을 때, 바운디드 컨텍스트는 물리적 경계가 되고 각 Subdomain은 논리적 경계(namespace/module/package)가 된다.

### 실생활 속 Bounded Context
토마토 예시: 식물학에서는 꽃에서 자라 씨를 맺으므로 과일이지만, 요리에서는 질감·맛을 기준으로 채소로 분류되고, 세금(1893년 미국 대법원 판례)에서도 관세 목적상 채소로 분류됐다. 같은 대상이라도 컨텍스트(분류 목적)가 다르면 적용되는 정의 자체가 달라진다.
냉장고 사례: 종이 박스는 냉장고의 가로·깊이가 문을 통과하는지 확인하는 모델, 줄자는 높이가 문틀에 맞는지 확인하는 모델이다. 각자의 좁은 목적에는 충분히 쓸모 있었기 때문에 더 정밀한 3D 모델은 오버엔지니어링이었을 것이다.
이 예시들은 완벽하게 현실을 복제하는 단 하나의 모델 대신, 각자의 문제에 맞게 충분히 쓸모 있는 더 단순한 여러 모델을 따로 쓰는 것이 더 효과적임을 보여준다 — 바운디드 컨텍스트로 시스템을 나누는 DDD 접근과 같은 원리다.

## 요약

도메인 전문가들의 정신 모델이 부서마다 달라 유비쿼터스 언어가 조직 전체에서 일관될 수 없을 때, 이를 여러 개의 바운디드 컨텍스트로 나누어 각 컨텍스트 안에서만 일관성을 보장하는 것이 DDD의 해법이다. Subdomain은 비즈니스 전략에 의해 발견되는 반면 Bounded Context는 소프트웨어 설계 결정으로서 직접 설계된다는 점에서 둘은 구분되며, 바운디드 컨텍스트는 모델·물리적 구현·팀 소유권이라는 세 가지 경계를 동시에 만든다.

---

## 밑줄 친 문장

> *"It's crucial to remember that subdomains are discovered and bounded contexts are designed."*

이 챕터 전체의 핵심을 한 문장으로 압축한 문장. Subdomain은 비즈니스가 이미 정해놓은 것을 분석해서 찾아내는 것이고, Bounded Context는 소프트웨어 설계 결정으로 직접 긋는 경계라는 뜻.

> *"Architectural design is system design. System design is contextual design—it is inherently about boundaries (what's in, what's out, what spans, what moves between), and about trade-offs."*
> — Ruth Malan

아키텍처 설계는 결국 무엇을 안에 두고 무엇을 밖에 둘지 정하는 경계 설정의 문제라는 뜻. 바운디드 컨텍스트가 바로 그 경계를 정의하는 DDD의 도구다.
