# Chapter 3. Managing Domain Complexity

## 핵심 개념 (질문에 답하며 채워보세요)

### Inconsistent Models (모델의 불일치)
- 책의 텔레마케팅 회사 예시에서, '리드(lead)'라는 용어는 마케팅 부서와 영업 부서에서 각각 어떤 의미로 쓰이나요?
- 같은 용어가 부서마다 다른 의미로 쓰일 때, 사람 간 대화에서는 큰 문제가 안 되는데 왜 소프트웨어로 구현할 때는 문제가 될까요?
- 책에서 제시한 두 가지 잘못된 해결책은 무엇이고, 각각 왜 효과적이지 않나요?
  1. 모든 것을 아우르는 거대한 통합 모델(전사 ERD)
  2. 용어에 접두사를 붙이는 방법 (예: "marketing lead", "sales lead")

### Bounded Context (바운디드 컨텍스트)
- 바운디드 컨텍스트의 정의를 본인 말로 적어보세요. (힌트: 유비쿼터스 언어를 어떻게 나누나요?)
- "모델은 경계 없이는 존재할 수 없다 — 경계가 없으면 현실의 복사본이 되어버린다"는 문장이 의미하는 바는 무엇인가요?
- 유비쿼터스 언어가 "ubiquitous(보편적)"하다는 게 정확히 무슨 뜻인가요? 조직 전체에서 통용된다는 뜻일까요, 아니면 다른 의미일까요?

### Bounded Context의 크기 (Scope)
- 바운디드 컨텍스트가 넓을 때와 좁을 때 각각 어떤 트레이드오프가 있나요?
- "크기 자체는 중요하지 않다, 유용성이 중요하다"는 말은 무슨 뜻인가요?

### Bounded Context vs Subdomain
- 이 챕터에서 가장 중요한 구분입니다: "Subdomain은 발견되고(discovered), Bounded Context는 설계된다(designed)"는 문장을 본인 말로 설명해보세요.
- 둘 다 비즈니스 도메인을 더 작은 단위로 나누는 방법인데, 왜 두 개념이 모두 필요한가요?
- Bounded Context와 Subdomain은 항상 1:1로 매칭되나요? 그렇지 않다면 어떤 경우들이 있나요?

### Boundaries: Physical & Ownership
- 바운디드 컨텍스트가 만드는 두 가지 경계는 무엇인가요?
- 한 팀이 여러 바운디드 컨텍스트를 소유할 수 있나요? 반대로 한 바운디드 컨텍스트를 여러 팀이 소유할 수 있나요?
- 바운디드 컨텍스트 안에 여러 Subdomain이 있을 때, Subdomain은 어떤 경계가 되나요? (물리적 경계 vs 논리적 경계)

### 실생활 속 Bounded Context
- 토마토(tomato) 예시에서, 식물학·요리·세금(미국 1893년 판례) 각 맥락에서 토마토는 과일인가요 채소인가요? 왜 다른가요?
- 냉장고 사례에서, 종이 박스(cardboard)와 줄자(tape measure)는 각각 무엇을 모델링하나요? 왜 3D 모델을 만들지 않았나요?
- 이 예시들이 공통적으로 보여주는 것은 무엇인가요? (2장의 "All models are wrong, but some are useful"와 연결지어 생각해보세요)
