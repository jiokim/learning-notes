# Chapter 5. Implementing Simple Business Logic

## 핵심 개념

### 비즈니스 로직 구현 패턴 개요
비즈니스 로직은 소프트웨어가 만들어지는 이유 자체다. UI가 멋지고 DB가 빨라도, 소프트웨어가 비즈니스에 쓸모없다면 값비싼 기술 데모일 뿐이다. 2장에서 배웠듯 서브도메인마다 전략적 중요도와 복잡도가 다르므로, 모든 서브도메인에 같은 구현 패턴을 쓰는 건 비효율적이다 — 단순한 로직에 너무 정교한 패턴을 쓰면 불필요한(accidental) 복잡도가 생기고, 복잡한 로직에 너무 단순한 패턴을 쓰면 로직이 여기저기 중복되고 일관성이 깨진다. 그래서 이 챕터는 단순한 로직용 패턴(Transaction Script, Active Record)부터 시작해, 다음 챕터들에서 점점 복잡한 로직용 패턴(Domain Model 등)으로 나아간다.

### 트랜잭션 스크립트 (Transaction Script)
이 패턴은 시스템의 public interface를 소비자(consumer)가 실행할 수 있는 비즈니스 트랜잭션들의 모음으로 보고, 각 프로시저가 절차적으로 비즈니스 로직을 구현한다. 각 프로시저가 충족해야 할 단 하나의 요구사항은 트랜잭션적 동작(transactional behavior)이다 — 연산은 성공하거나 실패해야 하며, 실패 시 변경을 롤백하거나 보상 조치(compensating actions)를 실행해 시스템을 항상 일관된 상태로 유지해야 한다.
패턴 자체는 단순하지만, 저자가 디버깅한 수많은 프로덕션 이슈가 바로 이 트랜잭션적 동작을 잘못 구현한 데서 비롯됐을 만큼 가장 잘못 구현하기 쉬운 패턴이다. 대표적인 실패 사례로 ① 여러 업데이트를 트랜잭션 없이 따로 실행하는 경우(관계형 DB라면 트랜잭션으로 묶어 해결), ② DB 업데이트와 메시지 버스 publish처럼 서로 다른 저장 메커니즘에 걸친 분산 트랜잭션(복잡하고 오류가 나기 쉬워 쉽게 고쳐지지 않음 — 8장의 CQRS, 9장의 outbox 패턴이 다룸), ③ `void` 메서드라도 성공/실패를 호출자에게 암묵적으로 통신하기 때문에 생기는 암시적 분산 트랜잭션(재시도 시 중복 처리 위험 — 작업을 멱등하게 만들거나 낙관적 동시성 제어로 해결)이 있다.
이 패턴은 정의상 로직이 단순한 supporting subdomain, ETL, 외부 시스템 통합의 어댑터나 anticorruption layer의 일부로 자연스럽게 쓰인다. 다만 로직이 복잡해질수록 트랜잭션 간 중복이 생기고 일관성이 깨지기 쉬워, core subdomain에는 절대 쓰면 안 된다. 단순함 때문에 안티패턴으로 취급되기도 하지만, 이후에 배울 모든 비즈니스 로직 구현 패턴이 어떤 형태로든 이 패턴을 기반으로 하기 때문에 여전히 핵심적인 토대다.

### 액티브 레코드 (Active Record)
Transaction Script처럼 단순한 비즈니스 로직을 다루지만, 평평한(flat) 레코드가 아니라 일대다·다대다 관계를 가진 더 복잡한 데이터 구조를 다룬다. 각 데이터 구조 자체가 CRUD 같은 데이터 접근 로직을 스스로 구현하기 때문에 "active"라는 이름이 붙었다. 퍼시스턴스 책임 외에 필드 값 검증이나 비즈니스 절차를 포함할 수도 있지만, 데이터 구조와 행동(behavior, 비즈니스 로직)이 분리돼 있다는 게 이 패턴의 특징적 성질이다(보통 필드에 public getter/setter가 있다).
Active Record는 본질적으로 DB 접근을 최적화한 Transaction Script이므로, 사용자 입력 검증 정도가 최대인 CRUD 연산까지만 지원할 수 있다. "anemic domain model 안티패턴"이라는 비판이 있지만, 저자는 패턴을 도구로 본다 — 도구는 잘못된 맥락에 쓰일 때만 해를 끼치며, 단순한 로직에 더 정교한 패턴을 쓰는 것도 마찬가지로 불필요한 복잡도를 만든다. (참고로 책에서 "Active Record"는 디자인 패턴 자체를 가리키며, Rails의 Active Record 같은 구체적 프레임워크를 가리키는 게 아니다 — 패턴은 Martin Fowler의 책에서 먼저 명명됐고, 프레임워크는 그 구현 방법 중 하나로 나중에 등장했다.)

### 실용적인 태도 (Be Pragmatic)
대량의 IoT 이벤트 수집처럼 규모가 매우 큰 경우, 데이터 일관성을 다소 완화하는 것도 정당화될 수 있다. 핵심 질문은 "이벤트의 0.001%가 중복되거나 손실돼도 정말 큰 문제인가?"이며, 그 답은 보편적 법칙이 아니라 해당 비즈니스 도메인에서 그 손실이 실제로 어떤 위험과 영향을 미치는지에 따라 달라진다.

### 결론
Transaction Script와 Active Record 모두 supporting subdomain에 가장 잘 맞는다 — 전자는 ETL 같은 단순한 절차적 작업에, 후자는 더 복잡한 데이터 구조를 다루는 작업에 적합하다. 다음 챕터에서는 더 복잡한 비즈니스 로직을 다루는 domain model 패턴으로 넘어간다.

---

## 요약

비즈니스 로직이 단순한 경우, Transaction Script는 트랜잭션적 동작(성공·실패 또는 롤백·보상 조치)을 보장하는 절차로 로직을 구조화하고, Active Record는 거기에 더해 더 복잡한 데이터 구조에 대한 CRUD 접근을 캡슐화하지만, 두 패턴 모두 단순한 로직(주로 supporting/generic subdomain)에만 적합하며 core subdomain에는 절대 쓰면 안 된다.

---

## 밑줄 친 문장

> *"This pattern is a tool. Like any tool, it can solve problems, but it can potentially introduce more harm than good when applied in the wrong context."*

Active Record가 "anemic domain model 안티패턴"이라는 비판을 받는 것에 대한 저자의 반론을 압축한 문장. 패턴 자체에 좋고 나쁨이 있는 게 아니라, 맥락에 맞지 않게 적용될 때만 해가 된다는 관점은 이후 챕터에서 다른 패턴을 고를 때도 똑같이 적용된다.
