# Chapter 4. Integrating Bounded Contexts

## 핵심 개념 (질문에 답하며 채워보세요)

### 통합과 Contracts
- 바운디드 컨텍스트가 독립적으로 진화할 수 있다는 게, 서로 완전히 독립적이라는 뜻인가요? 왜 항상 통합 지점(contract)이 생길 수밖에 없나요?
- 왜 두 바운디드 컨텍스트 사이의 통합이 까다로운가요? (힌트: 두 컨텍스트는 같은 유비쿼터스 언어를 쓰고 있나요?)
- 이 챕터는 팀 간 협업의 성격에 따라 통합 패턴을 세 그룹으로 나눠요. 그 세 그룹은 무엇인가요?

### Cooperation: Partnership
- Partnership 패턴에서 통합은 어떤 방식(공식적 계약 vs 즉흥적 조율)으로 이루어지나요?
- 이 패턴이 효과적으로 동작하려면 팀 사이에 어떤 조건(협업 수준, 동기화 빈도 등)이 필요한가요?
- 왜 지리적으로 분산된 팀에는 이 패턴이 잘 안 맞을까요?

### Cooperation: Shared Kernel
- Shared Kernel은 정확히 무엇을 공유하나요? 왜 공유 범위를 최소한으로 유지해야 하나요?
- Shared Kernel 패턴은 "바운디드 컨텍스트는 한 팀이 소유해야 한다"는 3장의 원칙과 모순된다고 책은 말해요. 그런데도 이 패턴이 정당화될 수 있는 경우들은 무엇인가요?
- Shared Kernel을 적용할지 판단하는 핵심 기준은 무엇인가요? (힌트: 두 가지 비용을 비교해요)
- 모델이 자주 바뀔수록 어떤 비용이 커지고, 그래서 Shared Kernel은 어떤 종류의 서브도메인에 자연스럽게 적용되나요?

### Customer–Supplier: 기본 구조
- 이 관계에서 "upstream(공급자)"과 "downstream(소비자)"은 각각 무엇을 의미하나요?
- Cooperation 패턴과 달리, Customer-Supplier 관계에는 보통 어떤 비대칭(불균형)이 존재하나요?

### Customer–Supplier: Conformist
- Conformist 패턴은 언제 나타나나요? 다운스트림 팀이 업스트림 모델을 그대로 받아들이는 게 정당화될 수 있는 경우는 어떤 것들인가요?

### Customer–Supplier: Anticorruption Layer
- Anticorruption Layer(ACL)는 Conformist와 무엇이 다른가요? (힌트: 둘 다 권력은 업스트림에 있지만, 다운스트림의 선택이 달라요)
- 책에서 ACL을 쓰는 게 좋다고 말하는 세 가지 상황은 무엇인가요?
- ACL은 다운스트림의 유비쿼터스 언어를 어떻게 보호해주나요?

### Customer–Supplier: Open-Host Service
- Open-Host Service는 권력이 어느 쪽으로 기울어진 관계에서 쓰이나요?
- "published language"란 무엇이며, 왜 공급자의 내부 유비쿼터스 언어와는 다른가요?
- 이 패턴이 "ACL의 반대 방향"이라고 불리는 이유는 무엇인가요?
- 공급자가 동시에 여러 버전의 published language를 노출할 수 있다는 게 왜 유용한가요?

### Separate Ways
- 협력하지 않기로 하는 것도 하나의 통합 "패턴"이에요. 책이 제시하는 세 가지 이유는 무엇인가요? (소통 문제 / ○○ 서브도메인 / 모델 차이)
- 로깅 프레임워크 같은 제너릭 서브도메인 예시에서, 왜 서비스로 통합하기보다 각자 따로 구현하는 게 더 합리적인가요?
- Separate Ways를 절대 쓰면 안 되는 경우는 언제이고, 왜일까요?

### Context Map
- Context Map은 무엇을 시각화한 것인가요?
- Context Map이 알려줄 수 있는 세 가지 차원의 통찰은 무엇인가요? (힌트: 설계, 소통, 조직)
- Context Map 관리·유지에서 왜 "공동 책임(shared effort)"이 중요한가요?
- 하나의 바운디드 컨텍스트 안에서도 여러 통합 패턴이 동시에 쓰일 수 있다고 해요. 왜 그런 일이 생길 수 있나요?
